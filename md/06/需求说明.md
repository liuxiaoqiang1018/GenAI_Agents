# 第6课 - ATLAS 多智能体协调系统 需求说明

## 一、产品定位

面向大学生的 **AI 学习助手**，通过多个专业 Agent 协作，提供学习规划、笔记生成和学习建议服务。

## 二、用户角色

- **目标用户**：大学生（demo 中模拟为大三计算机专业学生"张明"）
- **用户数据**：
  - 个人档案（年级、专业、学习风格、擅长/薄弱科目、学习偏好）
  - 课程日程表（周一到周五的课程和活动）
  - 待办任务列表（作业、实验报告、考试复习，含截止日期和优先级）

## 三、功能需求

### 3.1 协调器（Coordinator）

| 需求 | 说明 |
|------|------|
| 意图识别 | 判断用户请求是否与学术/学习相关 |
| Agent 调度 | 学术相关时，分析需要哪些 Agent 参与处理 |
| 直接回答 | 非学术问题（闲聊、问天气等）直接回答，不走 Agent 流程 |

### 3.2 计划者 Agent（Planner）

| 需求 | 说明 |
|------|------|
| 日程感知 | 读取学生现有日程，避免时间冲突 |
| 优先级排序 | 高优先级任务优先安排 |
| 个性化 | 根据学生最佳学习时段和专注时长安排 |
| 输出格式 | 按天排列的具体时间安排表 |

### 3.3 笔记员 Agent（NoteWriter）

| 需求 | 说明 |
|------|------|
| 风格适配 | 根据学习风格生成材料（视觉型→思维导图/列表） |
| 结构化 | 生成有层次的学习笔记 |
| 输出格式 | 结构化的复习材料/知识总结 |

### 3.4 顾问 Agent（Advisor）

| 需求 | 说明 |
|------|------|
| 个性化建议 | 根据学生薄弱科目给出针对性建议 |
| 方法推荐 | 推荐适合该学生的学习方法 |
| 输出格式 | 短期建议 + 长期改进方向 |

### 3.5 汇总输出

| 需求 | 说明 |
|------|------|
| 结果合并 | 将所有被激活的 Agent 输出合并展示 |
| 分区展示 | 按【学习计划】【学习材料】【学习建议】分区 |

## 四、流程需求

```
用户输入
  │
  ▼
协调器判断 ──── 非学术 ──→ 直接回答 → 结束
  │
  │ 学术相关
  ▼
档案分析（提取学习特征）
  │
  ├── 条件路由（按需激活）
  │
  ▼           ▼           ▼
计划者      笔记员       顾问      ← 可并行，不一定全部激活
  │           │           │
  └───────────┼───────────┘
              ▼
          汇总输出 → 结束
```

关键流程规则：
1. **不是所有 Agent 都参与每次请求** — 协调器按需激活
2. **被激活的 Agent 可以并行执行** — 互不依赖
3. **非学术问题短路返回** — 不浪费 Agent 调用

## 五、数据需求

| 数据 | 来源 | 当前实现 |
|------|------|----------|
| 学生档案 | 应为用户注册时填写 | 硬编码 `STUDENT_PROFILE` |
| 课程日程 | 应对接教务系统 | 硬编码 `STUDENT_CALENDAR` |
| 待办任务 | 应为用户手动添加 | 硬编码 `STUDENT_TASKS` |

## 六、技术约束

| 约束 | 说明 |
|------|------|
| LLM 调用次数 | 最少 1 次（非学术直接回答），最多 5 次（协调器+档案+3个Agent） |
| 协调器输出格式 | 必须返回可解析的 JSON，代码需做容错处理 |
| 状态管理 | 多 Agent 并行返回的结果需要深度合并（dict_reducer） |

## 七、与简单 Agent 的区别

| 维度 | 单 Agent（第2课） | 本课多智能体 |
|------|-------------------|-------------|
| 决策者 | LLM 自己决定下一步 | **协调器决定调度哪些 Agent** |
| 专业性 | 一个 Agent 做所有事 | **每个 Agent 只做一件事** |
| 可扩展性 | 加功能要改 Agent | **加一个新 Agent 节点即可** |
| LLM 调用 | 循环直到完成 | **并行调用，固定步骤** |
