# 第7课 - 科研论文助手（循环图 + 自我评估）

## 环境准备

```bash
# 激活虚拟环境后安装
pip install langgraph langchain-openai langchain-core python-dotenv
```

`main_debug.py` 只需要 `openai` 和 `python-dotenv`。

## 概述

本课构建一个**科研论文研究助手**，核心亮点是 LangGraph 的**循环图**：
- Agent 可以反复调用工具（搜索论文、下载PDF）直到收集够信息
- Judge 节点自我评估答案质量，不合格则打回重做

这是前几课没有出现的模式——**图中有环（Loop）**。

## 动机

前几课的图都是**无环的**（DAG）：
- 第4课：A→B→C，固定顺序
- 第6课：协调器→Agent们→汇总，有分支但不回头

但真实的 AI 应用经常需要**循环**：
- 搜索结果不够？再搜一次
- 答案质量不好？重新规划再来一轮
- 工具调用出错？重试

LangGraph 支持图中有环，这就是它比简单 Pipeline 强大的地方。

## 架构

```
用户提问
   │
   ▼
┌──────────┐
│  决策节点  │  判断是否需要研究（闲聊 vs 学术问题）
└──────────┘
   │         \
   │ 需要研究   不需要 → 直接回答 → 结束
   ▼
┌──────────┐
│  规划节点  │  制定分步研究计划
└──────────┘
   │
   ▼
┌──────────┐     ┌──────────┐
│  Agent   │ ←──→│  工具节点  │  ← 循环！Agent 调用工具，工具返回结果
│  节点    │     │（搜索/下载）│     Agent 再决定是否继续调用
└──────────┘     └──────────┘
   │
   │ 不再调用工具
   ▼
┌──────────┐
│  评审节点  │  自我评估答案质量
└──────────┘
   │         \
   │ 不合格    合格 → 结束
   ▼
  回到规划节点（循环！最多重试2次）
```

## 核心概念

### 1. 循环图（Cyclic Graph）

```python
# Agent ↔ Tools 循环
workflow.add_conditional_edges("agent", should_continue, {
    "continue": "tools",   # 还要调用工具 → 回到 tools
    "end": "judge",        # 不需要工具了 → 去评审
})
workflow.add_edge("tools", "agent")  # 工具执行完 → 回到 agent

# Judge → Planning 循环
workflow.add_conditional_edges("judge", final_answer_router, {
    "planning": "planning",  # 不合格 → 重新规划
    "end": END,              # 合格 → 结束
})
```

**Java 类比**：就像 `while` 循环 + `break` 条件。
```java
while (true) {
    AgentResult result = agent.run(state);
    if (!result.hasToolCalls()) break;  // 退出循环
    toolResults = executeTool(result.getToolCalls());
    state.addMessages(toolResults);
}
```

### 2. 结构化输出（Structured Output）

```python
# 强制 LLM 返回指定格式
decision_making_llm = base_llm.with_structured_output(DecisionMakingOutput)
```

让 LLM 返回 `{"requires_research": true, "answer": null}` 而不是自由文本。

**Java 类比**：像用 JSON Schema 约束 API 响应格式。

### 3. 自我评估（Self-Evaluation / Judge）

Agent 生成答案后，由另一个 LLM 调用（Judge）来评估质量：
- 是否直接回答了用户问题？
- 是否有引用来源？
- 是否考虑了之前的反馈？

不合格则带着反馈重新规划，最多重试2次。

**Java 类比**：像代码 Review 流程——开发者（Agent）提交代码，审查者（Judge）审核，不通过则打回修改。

### 4. 工具调用循环（Tool Calling Loop）

这是 LLM + 工具的标准模式：
1. LLM 决定调用哪个工具（`tool_calls`）
2. 执行工具，把结果放回消息
3. LLM 再看结果，决定是否继续调用
4. 重复直到 LLM 不再调用工具

## 与前几课的对比

| 维度 | 第4课（线性） | 第6课（分支） | 本课（循环） |
|------|-------------|-------------|-------------|
| 图结构 | A→B→C | 协调器→分支→汇总 | **有环！可以回头** |
| 工具调用 | 无 | 无 | **Agent↔Tools 循环** |
| 质量控制 | 无 | 无 | **Judge 自我评估** |
| 重试机制 | 无 | 无 | **不合格打回重做** |
| LLM 调用次数 | 固定3次 | 动态(2-5次) | **不确定（取决于循环次数）** |

## 关键洞察

1. **循环是让 Agent "自主"的关键**：没有循环，Agent 只能走预设路径；有了循环，Agent 可以根据中间结果动态决定下一步
2. **自我评估防止低质量输出**：Judge 节点就像一个质量门（Quality Gate），确保输出达标才放行
3. **最大重试次数防止死循环**：`num_feedback_requests >= 2` 就强制结束，这是工程实践中的重要保护
4. **结构化输出让路由可靠**：用 `with_structured_output()` 而不是正则解析 JSON，更稳定
