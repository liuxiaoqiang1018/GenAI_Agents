# 第9课 - 智能客服路由系统（多条件路由）

## 环境准备

```bash
pip install langgraph langchain-core httpx python-dotenv
```

`main_debug.py` 只需要 `httpx` 和 `python-dotenv`。

## 概述

本课构建一个**智能客服路由系统**，处理流程：
1. **分类**（Categorize）：把工单归类为 技术问题 / 账单问题 / 一般咨询
2. **情感分析**（Sentiment）：判断用户情绪 积极 / 中性 / 消极
3. **条件路由**（Route）：根据 情感 × 类别 走不同分支
4. **处理/升级**：消极情绪 → 直接升级人工；其他 → 对应专业客服处理

## 动机

这是一个典型的**企业客服场景**：
- 用户投诉语气激烈 → 别让 AI 回复，直接转人工（防止火上浇油）
- 技术问题 → 走技术客服模板
- 账单问题 → 走财务客服模板
- 一般咨询 → 走通用客服模板

核心模式：**先分析，再路由，分而治之**。

## 核心概念

### 1. 多条件路由（Multi-Condition Routing）

```python
def route_query(state):
    if state["sentiment"] == "消极":
        return "escalate"           # 情绪差 → 优先升级，不管什么类别
    elif state["category"] == "技术":
        return "handle_technical"
    elif state["category"] == "账单":
        return "handle_billing"
    else:
        return "handle_general"
```

注意路由的**优先级**：情感优先于类别。即使是简单的账单问题，如果用户很生气，也转人工。

**Java 类比**：
```java
// Spring Cloud Gateway 的路由配置
if (sentiment.equals("消极")) return "human-agent-service";
switch (category) {
    case "技术": return "tech-support-service";
    case "账单": return "billing-service";
    default: return "general-service";
}
```

### 2. 升级模式（Escalation Pattern）

消极情绪的工单不走 LLM，直接返回固定文案 + 转人工。这是**安全兜底**——避免 AI 在用户情绪激动时说出不当回复。

**Java 类比**：像熔断器（Circuit Breaker）模式——检测到异常就立即降级处理。

### 3. 与第4课的对比

| 维度 | 第4课（线性流水线） | 本课（路由系统） |
|------|-------------------|-----------------|
| 图结构 | A→B→C 固定 | A→B→**条件分支**→C1/C2/C3/C4 |
| 路由 | 无 | **4条路径，优先级判断** |
| 处理节点 | 每个都执行 | **只执行匹配的那个** |
| 实际应用 | 文本分析 | **客服/工单系统** |

## 关键洞察

1. **分析和处理分离**：分类+情感分析是"分析层"，handle_xxx 是"处理层"，路由函数是"决策层"。这种分层和微服务架构一样
2. **优先级路由**：消极情绪优先于类别——这是业务逻辑，不是技术约束
3. **升级不用 LLM**：`escalate()` 直接返回固定文案，不调 LLM——最关键的路径反而最简单
4. **容易扩展**：加一个"紧急"类别，只需加一个节点+一条路由规则
