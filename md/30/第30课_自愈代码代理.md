# 第30课：自愈代码代理（Self-Healing Code）

## 课程概述

本课构建一个**自动修复代码错误的 Agent 系统**：传入一个有 Bug 的函数和参数，Agent 自动执行 → 捕获错误 → 生成 Bug 报告 → 搜索历史 Bug 记忆 → LLM 生成修复方案 → 热补丁代码 → 再次执行验证，形成**自愈循环**。

**核心模式**：自愈循环 + 向量数据库记忆（ChromaDB）+ 动态代码热补丁（exec）

## 架构流程

```
代码执行 → [条件路由]
  ├─ 无错误 → 结束
  └─ 有错误 → 生成Bug报告 → 搜索历史记忆(ChromaDB)
                                    ↓
                            [有相似记忆？]
                            ├─ 有 → 过滤记忆 → [需更新？]
                            │                   ├─ 有 → 更新记忆 → (循环直到更新完)
                            │                   └─ 无 → 保存新记忆
                            └─ 无 → 保存新记忆
                                    ↓
                            LLM生成修复代码 → 热补丁(exec) → 代码执行(循环回去)
```

**关键循环**：代码执行 → 错误 → 修复 → 再执行 → 直到成功

## 关键概念标注

| 概念 | 在代码中的位置 | Java 类比 |
|---|---|---|
| **State（状态）** | `State(BaseModel)` — 含函数引用、错误信息、Bug报告、记忆搜索结果 | POJO + 函数引用 |
| **代码执行节点** | `code_execution_node()` — try/except 捕获任意函数的错误 | 反射调用 + try/catch |
| **Bug报告生成** | `bug_report_node()` — LLM 生成结构化 Bug 报告 | 异常分析器 |
| **向量记忆搜索** | `memory_search_node()` — ChromaDB 查询相似 Bug | 全文检索 / ES |
| **记忆过滤** | `memory_filter_node()` — 距离 < 0.3 才认为相关 | 相似度阈值过滤 |
| **记忆更新/新增** | `memory_modification_node()` / `memory_generation_node()` | CRUD |
| **代码修复** | `code_update_node()` — LLM 生成修复后的函数代码 | 代码生成 |
| **热补丁** | `code_patching_node()` — `exec()` 动态替换函数 | 热部署 / 字节码替换 |
| **多条件路由** | 4 个 router 函数控制图的分支 | 策略模式链 |

## 本课特点（对比之前课程）

1. **向量数据库（ChromaDB）**：第一次引入向量存储，用于 Bug 报告的相似性搜索和记忆管理
2. **动态代码执行（exec）**：`exec()` 热替换函数，Python 独有的动态能力
3. **自愈循环**：修复后自动重新执行验证，失败则继续修复（类似第7课的自我评估循环）
4. **Callable 状态**：State 中存储函数引用（`Callable`），不只是数据
5. **4 个条件路由**：本课路由逻辑最复杂，4 个 router 控制 8 个节点的流转

## 与简单代理的对比

| 维度 | 简单 Agent | 本课自愈代码系统 |
|---|---|---|
| 输入 | 用户文本 | 函数 + 参数 |
| 输出 | 文本回答 | 修复后的函数代码 |
| 记忆 | 无或对话记忆 | 向量数据库持久化 Bug 记忆 |
| 循环 | 单次 | 执行→修复→再执行（直到成功） |
| 路由 | 1-2 个 | 4 个条件路由 |
