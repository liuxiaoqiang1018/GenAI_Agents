## 概述

本课构建一个**AI项目管理助手**，自动完成项目规划的完整流程：

1. **任务生成**：从项目描述中提取可执行的任务列表
2. **依赖分析**：识别任务之间的依赖关系
3. **任务排期**：根据依赖关系安排开始/结束日期
4. **人员分配**：根据团队成员技能分配任务
5. **风险评估**：评估每个任务的风险并计算总分
6. **洞察生成**：生成改进建议（如果风险过高，回到第3步重新排期）

核心模式：**自反思循环** —— 评估风险后，如果不满意就生成洞察、重新规划，直到风险降低或达到最大迭代次数。

## 动机

项目规划是一个需要反复调整的过程：
- 第一版排期可能有瓶颈
- 第一次人员分配可能不均衡
- 风险评估发现问题后，需要**回头调整**

这就是**自反思**（Self-Reflection）模式的价值：Agent不是"做一次就交差"，而是"做完后检查，不好就改"。

## 核心概念

### 1. 自反思循环（Self-Reflection Loop）

风险评估后的路由逻辑：

```python
def router(state):
    if iteration >= max_iteration:
        return END                    # 达到上限，强制结束
    if len(risk_scores) > 1:
        if risk_scores[-1] < risk_scores[0]:
            return END                # 风险降低了，结束
        else:
            return "insight_generator" # 风险没降，继续优化
    else:
        return "insight_generator"     # 第一轮，至少做两次
```

**Java 类比**：
```java
// 像 Spring Retry + 质量门禁
@Retryable(maxAttempts = 3, condition = "riskScore >= threshold")
public ProjectPlan plan(String description, Team team) {
    Schedule schedule = scheduler.schedule(tasks, insights);
    Allocation allocation = allocator.assign(tasks, team, insights);
    int riskScore = riskAssessor.assess(schedule, allocation);
    // 如果风险高，自动重试（带上洞察反馈）
}
```

### 2. 结构化输出（Structured Output）

原始 notebook 用 `llm.with_structured_output(TaskList)` 让 LLM 直接返回 Pydantic 对象。我们用 JSON 格式约束 + 解析实现同样效果：

```python
prompt = "...返回JSON格式：{\"tasks\": [{\"name\": \"...\", \"days\": 3}]}"
response = call_llm(prompt)
tasks = json.loads(response)  # 解析为dict
```

### 3. 迭代状态累积

每次迭代的排期、分配、风险都保存到 `_iteration` 列表中，下一轮可以参考上一轮的结果：

```python
state["schedule_iteration"].append(schedule)       # 每轮排期都存着
state["risk_score_iterations"].append(risk_score)   # 风险分数变化趋势
```

这让 LLM 在下一轮生成洞察时能看到"之前做了什么、哪里出了问题"。

## 关键洞察

1. **自反思 = 做完后自己检查**：风险评估就是"检查"，洞察生成就是"反思"，重新排期就是"改进"
2. **至少做两次**：第一轮必须进入洞察生成，确保有对比基准。像代码 Review 一样，至少看两遍
3. **结构化输出是关键**：项目管理需要精确的任务列表、日期、人员分配，不能是随意文本
4. **洞察是桥梁**：洞察生成节点把"发现的问题"转化为"下一轮的改进方向"，传递给排期和分配节点
