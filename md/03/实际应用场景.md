# 第3课：数据分析代理在实际应用中的流程

## Demo vs 实际应用的对比

### Demo 中的流程

```
代码里写死 1000 条假数据（create_sample_data）
    → 用户在终端输入问题
    → Agent 生成查询 → 执行 → 返回文本回答
```

### 实际应用中的流程

```
用户上传 Excel / 连接数据库 / 对接业务系统 API
    → 用户在网页/钉钉/企微中提问
    → Agent 生成查询 → 执行 → 返回回答（可能带图表）
```

---

## 实际场景举例

### 场景1：销售总监看报表

**传统方式：**
销售总监想知道"上个月华东区哪个产品卖得最好"，需要：
1. 找数据分析师提需求
2. 分析师写 SQL / 拉 Excel 透视表
3. 等半天拿到结果

**用数据分析 Agent：**
销售总监直接在企业聊天窗口问："上个月华东区销量最高的产品是哪个？"
- Agent 自动生成 `df[df['区域']=='华东']['产品'].value_counts().head(1)`
- 几秒钟返回答案

### 场景2：运营人员做日报

运营每天早上要看昨日数据，以前手动打开 BI 系统拉数。现在直接问：
- "昨天的 DAU 是多少？"
- "和上周同期比怎么样？"
- "哪个渠道的新增用户最多？"

Agent 自动查询、对比、生成结论。

### 场景3：客服查订单

客服接到用户电话，需要查订单状态。以前要切到后台系统搜索。现在：
- "帮我查一下订单号 20240315001 的状态"
- Agent 生成查询，返回"该订单已发货，快递单号 SF1234567890，预计明天到达"

### Demo 对应实际应用的映射

| Demo 中 | 实际应用中 |

| `create_sample_data()` 生成假数据 | 连接 MySQL / 读取 Excel / 调用业务 API |
| `pd.eval()` 执行查询 | 执行 SQL / Pandas 操作 / API 调用 |
| `ModelRetry` 重试 | 同样需要，LLM 生成的 SQL 也可能写错 |
| 终端 `input()` 输入 | 网页输入框 / 聊天机器人 / API 请求 |

## 实际应用中要多考虑的事情

### 1. 安全性（最重要）

Demo 用的 `pd.eval()` 已经比直接 `exec()` 安全，但实际应用中：
- **SQL 注入防护**：Agent 生成的 SQL 必须是只读的（只允许 SELECT）
- **权限控制**：不同用户只能查自己有权限的数据
- **敏感数据脱敏**：薪资、身份证号等字段需要遮盖

```python
# 实际应用中的工具会多一层权限检查
def sql_query(ctx, query: str) -> str:
    # 禁止危险操作
    if any(keyword in query.upper() for keyword in ['DELETE', 'DROP', 'UPDATE', 'INSERT']):
        raise ModelRetry('只允许 SELECT 查询')

    # 检查用户权限
    if '薪资' in query and ctx.deps.user_role != 'hr':
        raise ModelRetry('你没有查看薪资数据的权限')

    return db.execute(query)
```

### 2. 数据量

Demo 只有 1000 行，实际可能是千万行。解决方案：
- 不把整个表加载到内存，而是生成 SQL 让数据库执行
- 对大表做预聚合（每天定时计算汇总表）
- 限制查询结果行数（`LIMIT 1000`）

### 3. 上下文信息

Demo 的 System Prompt 很简单。实际应用中需要告诉 Agent：
- 有哪些表、每个字段的含义
- 业务术语定义（"活跃用户"的标准是什么）
- 常见查询模板

### 4. 结果展示

Demo 只返回文本。实际应用中用户往往还需要：
- 图表（"画一个柱状图"）
- 导出 Excel
- 定时推送（"每天早上8点发日报"）

---

## 总结

这个 Demo 虽然只是对一个内存中的小 DataFrame 做查询，但它展示的**模式**是完全通用的：

> **用户用自然语言提问 → Agent 自动生成代码/查询 → 执行 → 出错就重试 → 成功就返回自然语言回答**

把 `pd.eval()` 换成 `db.execute(sql)` 就是一个数据库查询助手，换成 `api.call()` 就是一个业务系统助手。核心机制（工具调用 + 重试）不变。
