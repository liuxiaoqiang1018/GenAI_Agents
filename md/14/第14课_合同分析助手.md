# 第14课 - 合同分析助手（Map-Reduce 并行 + 多角色审查）

## 环境准备

```bash
pip install langgraph langchain-core httpx python-dotenv
```

`main_debug.py` 只需要 `httpx` 和 `python-dotenv`。

## 概述

本课构建一个**合同分析助手**，自动审查合同并生成报告。核心流程：

1. **合同分类** → 识别合同类型和行业
2. **条款检索** → 获取该类合同的标准条款列表
3. **条款审查**（并行）→ 对每个标准条款，检查合同中是否清晰体现（**Map**）
4. **生成审查计划** → 根据合同类型生成多角色审查步骤（**Reduce/汇聚后继续**）
5. **多角色审查**（并行）→ 每个法律角色独立审查合同（**Map**）
6. **生成最终报告** → 汇总所有修改建议（**Reduce**）

核心模式：**Map-Reduce** —— 把一个大任务拆成N个小任务并行执行，再汇总结果。

## 动机

合同审查需要：
- 逐条检查标准条款是否存在且清晰
- 从多个专业角度审查（劳动法、知识产权、合规、隐私等）

这些子任务之间**互不依赖**，完全可以并行执行。LangGraph 的 `Send()` 机制就是为此设计的。

## 核心概念

### 1. Map-Reduce 并行（Send 机制）

LangGraph 的 `Send()` 可以动态创建多个并行任务：

```python
from langgraph.constants import Send

def fan_out_clauses(state):
    # 为每个条款创建一个并行任务
    return [
        Send("check_clause", {"contract_text": state["contract_text"], "clause": c})
        for c in state["clauses"]
    ]

# 条件边：动态扇出
workflow.add_conditional_edges("retrieve_clauses", fan_out_clauses, ["check_clause"])
```

**Java 类比**：
```java
// 像 Java 的 CompletableFuture.allOf()
List<CompletableFuture<Result>> futures = clauses.stream()
    .map(clause -> CompletableFuture.supplyAsync(
        () -> checkClause(contract, clause)))
    .collect(toList());
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
```

### 2. 状态累积（Annotated + operator.add）

并行节点各自返回结果，通过 `operator.add` 自动合并到列表：

```python
class State(TypedDict):
    # Annotated[List, operator.add] 表示：多个节点返回的结果自动拼接
    clause_analysis: Annotated[List[str], operator.add]
    modifications: Annotated[List[dict], operator.add]
```

**Java 类比**：像 `ConcurrentLinkedQueue` —— 多线程各自往队列里放结果，最后一起取出。

### 3. 多角色审查（Multi-Role Review）

合同审查需要多个专业视角，每个角色是一个独立的 LLM 调用：
- 劳动法专家审查
- 知识产权顾问审查
- 合规官审查
- 数据隐私官审查

每个角色看同一份合同，但关注点不同，输出各自的修改建议。

### 4. 与前几课的对比

| 维度 | 第9课（多条件路由） | 第13课（自反思循环） | **第14课（Map-Reduce）** |

## 关键洞察

1. **Send() 就是动态 for 循环**：`main_debug.py` 里可以看到，Send 并行在单机上就是依次执行每个子任务
2. **operator.add 就是 list.extend()**：多个并行节点的返回值自动拼接到同一个列表
3. **两次 Map-Reduce**：先对条款做一次（检查完整性），再对角色做一次（多视角审查）
4. **简化了向量检索**：原始 notebook 用 Pinecone 检索条款，我们用内置的标准条款库代替
