# 第11课 - 旅行规划器（交互式输入 + LLM生成）

## 环境准备

```bash
pip install langgraph langchain-core httpx python-dotenv
```

`main_debug.py` 只需要 `httpx` 和 `python-dotenv`。

## 概述

本课构建一个**交互式旅行规划器**，流程：
1. **输入城市**：用户告诉系统想去哪个城市
2. **输入兴趣**：用户输入感兴趣的活动（美食、历史、购物等）
3. **生成行程**：LLM 根据城市+兴趣生成一日游行程

这是一个**纯线性流水线**，但重点是：节点里包含**用户交互输入**，不是所有数据都预设好的。

## 动机

旅行规划是一个典型的**收集信息 → 生成结果**场景：
- 先分步收集用户需求（城市、兴趣）
- 最后一步调用 LLM 生成个性化内容

这个模式在很多场景都适用：表单填写、问卷调查、配置向导等。

## 架构

```
用户请求
   │
   ▼
┌──────────┐
│ 输入城市  │  ← 用户交互输入
└──────────┘
   │
   ▼
┌──────────┐
│ 输入兴趣  │  ← 用户交互输入
└──────────┘
   │
   ▼
┌──────────┐
│ 生成行程  │  ← LLM 调用
└──────────┘
   │
   ▼
  结束
```

## 核心概念

### 1. 交互式节点（Interactive Nodes）

前两个节点不调用 LLM，而是等待用户输入：

```python
def input_city(state):
    city = input("请输入城市: ")
    return {"city": city}

def input_interests(state):
    interests = input("请输入兴趣: ")
    return {"interests": interests.split(",")}
```

**Java 类比**：
```java
// 像 Spring MVC 的多步表单
@GetMapping("/step1")  // 输入城市
@GetMapping("/step2")  // 输入兴趣
@PostMapping("/submit") // 生成行程
```

### 2. 状态累积（State Accumulation）

每个节点只负责填充自己的字段，状态逐步丰富：

```
step1: { city: "北京" }
step2: { city: "北京", interests: ["美食", "历史"] }
step3: { city: "北京", interests: ["美食", "历史"], itinerary: "..." }
```

**Java 类比**：像 Builder 模式 —— `new Trip().city("北京").interests("美食").build()`

### 3. 与前几课的对比

| 维度 | 第4课（线性流水线） | 第10课（条件终止） | **第11课（交互线性）** |
|------|-------------------|-------------------|---------------------|
| 图结构 | A→B→C 线性 | A→B?→C?→D?→E | A→B→C 线性 |
| 输入方式 | 预设数据 | 预设数据 | **用户交互输入** |
| LLM调用 | 每步都调 | 每步都调（可短路） | **只有最后一步调** |
| 核心模式 | 分析流水线 | 条件提前终止 | **收集→生成** |

## 关键洞察

1. **不是每个节点都需要LLM**：前两步是纯用户交互，只有最后一步调LLM。图节点 ≠ LLM调用
2. **状态就是上下文**：用户输入的城市和兴趣存在 State 里，最后一步可以直接使用
3. **线性不代表简单**：虽然图结构是直线，但"收集信息→生成结果"是非常实用的模式
4. **prompt 模板的价值**：把用户输入的变量（城市、兴趣）填入 prompt 模板，让 LLM 生成个性化内容
